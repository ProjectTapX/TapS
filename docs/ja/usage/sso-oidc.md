# シングルサインオン（SSO / OIDC）

TapS Panel は標準的な **OpenID Connect (OIDC)** ログインをサポートしています。Google、Microsoft Entra ID、Keycloak、Casdoor、Logto、または OIDC 仕様に準拠し `.well-known/openid-configuration` を提供する任意の IdP と互換性があります。

> このドキュメントでは、管理者による IdP 設定、ユーザーの紐づけ/解除、ログイン方式の切り替え、ロックアウト時の復旧について説明します。

---

## 1. 前提条件

1. **Panel が外部からアクセス可能なアドレスを持つ必要があります**。OIDC のコールバックは IdP からブラウザにリダイレクトされ、ブラウザが Panel に戻ります。IdP が内部ネットワーク/`localhost` 上の Panel に到達できなくても問題ありませんが、**ブラウザが Panel にアクセスできる**必要があります。
2. IdP で OIDC クライアントを事前に登録し、以下を準備してください：
   - **Issuer URL**（例：`https://accounts.google.com`、`https://login.microsoftonline.com/{tenantId}/v2.0`、セルフホスト `https://sso.example.com/realms/master`）
   - **Client ID** + **Client Secret**

---

## 2. 設定手順（管理者）

### 2.1 Panel の公開 URL を設定

**システム設定 → SSO → 「Panel 公開 URL」** に移動し、外部 URL（プロトコルを含む）を入力します。例：`https://taps.example.com`。**この設定がないと、すべての SSO フローが失敗します。**

コールバック URL は自動的に `<公開 URL>/api/oauth/callback/<プロバイダースラッグ>` となります。

### 2.2 プロバイダーの追加

**システム設定 → SSO → プロバイダーを追加** で、テンプレート（Google / Microsoft Entra / Keycloak / Logto / カスタム）を選択します。必須フィールド：

| Field | 説明 |
|---|---|
| Name (slug) | URL セーフな識別子。**保存後は変更不可**。コールバック URL のパスを決定します |
| Display Name | ログインページに表示されるボタンのテキスト |
| Issuer | OIDC の Issuer URL。`.well-known/openid-configuration` 経由でエンドポイントが自動検出されます |
| Client ID / Secret | IdP 登録時に取得した認証情報。Secret は AES-256-GCM で保存時に暗号化されます |
| Scopes | デフォルト `openid profile email`。必要に応じて追加してください |
| Allowed Email Domains | 空欄 = 制限なし。複数はカンマで区切ります |
| Auto-Create Account | オン = IdP のメールアドレスにローカルアカウントが一致しない場合、ローカルアカウントを自動作成。オフ = ログインを拒否 |
| Default Role | 「自動作成」が有効な場合のみ適用されます |

**Test Issuer** をクリックして到達性を確認し、保存します。

### 2.3 IdP でコールバック URL を登録

**モーダルのフッターに**プロバイダーのコールバック URL が表示されます。これを IdP コンソールの「Redirect URI」ホワイトリストにコピーしてください。

### 2.4 ログイン方式の切り替え

**システム設定 → SSO → ログイン方式** で、以下の 3 つから選択します：

- `password-only` — デフォルト。ログインページにはユーザー名/パスワードのみ表示
- `oidc+password` — 両方対応、**推奨**
- `oidc-only` — SSO のみ。パスワードログインは拒否

> **ロックアウト防止**：`oidc-only` への切り替えには「少なくとも 1 人の管理者が有効な SSO プロバイダーに紐づけられていること」が必要です。そうでなければバックエンドが拒否します。理由は明白で、IdP がダウンすると誰もログインできなくなるためです。

---

## 3. ユーザーの紐づけ / 解除

一般ユーザーは **アバター（右上） → アカウント設定** で紐づけ済みの SSO 一覧を確認できます。

- **紐づけ**：「X を紐づけ」をクリックすると SSO ログインフローで IdP にリダイレクトされます。IdP の承認後、Panel に戻り現在のアカウントに自動紐づけされます。
- **解除**：一覧行の「解除」をクリックします。`oidc-only` モードでは、最後の紐づけの解除は拒否されます（セルフロックアウト防止）。
- **自動紐づけ**：ユーザーが初めて SSO でログインした際、以下の順序でマッチングされます：
  1. `(provider, subject)` が既に存在 → 元のアカウントを特定
  2. IdP が返すメールアドレスがローカルアカウントの `email` と一致 → そのアカウントに紐づけ
  3. 一致なし → 「自動作成」が有効かどうかを確認

管理者はユーザー管理ページで任意のユーザーの紐づけを管理できます（ロックアウトされた同僚の紐づけ解除に便利です）。

---

## 4. ロックアウト後の復旧

不幸にも `oidc-only` に切り替えた後で IdP が壊れた場合、誰もログインページにアクセスできません。Panel ホストのシェルで以下を実行してください：

```bash
# パスワードログインに戻す（すべての SSO 設定は保持され、後で oidc+password に切り替え可能）
./taps-panel reset-auth-method --to password-only
```

Panel プロセスの再起動は不要です。次のログインリクエストで新しい値が読み取られます。

## 5. セキュリティに関する注意事項

- **State + PKCE + Nonce** がすべて有効です。State は HMAC-SHA256 で署名されます（provider + nonce + 有効期限、TTL 5 分）。**PKCE verifier は Panel プロセスのメモリに保存**されます（URL には含まれません）。TTL 10 分 + maxEntries 上限で DoS を防止します。
- **Client Secret** は保存時に AES-256-GCM で暗号化されます。鍵ファイルは `<dataDir>/secret-encryption.key`（初回起動時に自動生成、パーミッション 0600）にあります。**このファイルを紛失するとすべてのシークレットが無効になります**。バックアップに含めてください。
- **メールアドレスの大文字小文字正規化**：IdP が返すメールアドレスは入力時に `ToLower` されます。これにより、大文字小文字の違いを利用した管理者の自動紐づけ拒否のバイパスを防止します。
- **コールバックエラーコードマッピング**：すべてのコールバック失敗は `CallbackError{Code, Err}` 型ラッパーを使用します。URL フラグメントには安定したコード（例：`sso.token_exchange_failed`）のみが渡され、IdP の内部エラーがブラウザに漏洩することはありません。監査ログには完全な内部エラーが保持されます。
- **トークンのパススルー**：OIDC 成功後、Panel は独自の TapS JWT を発行し、URL ハッシュフラグメント（`#oauth-token=...`）経由で SPA に渡します。IdP のアクセス/リフレッシュトークンは**ブラウザに渡りません**。

## 6. FAQ

**Q: コールバックで `redirect_uri_mismatch` が繰り返し報告される場合は？**
A: IdP コンソールの「Redirect URI」が**完全に一致**しているか確認してください（プロトコル、ポート、`/api/oauth/callback/<slug>` を含む）。slug は変更できないため、最初から慎重に選択してください。

**Q: 1 つのローカルアカウントに複数の IdP を紐づけられますか？**
A: はい。アカウント設定ページで同じアカウントに複数の IdP を紐づけることができます。

**Q: ユーザーが IdP でメールアドレスを変更した場合、自動同期されますか？**
A: ログイン時に最新のメールアドレスが `sso_identities.email` に書き戻されますが、ローカルの `users.email` は**変更されません**（ユーザー管理ページとの競合を回避するため）。

**Q: SSO プロバイダーを削除するとすべての紐づけが削除されますか？**
A: はい。削除はカスケードし、その `provider_id` の `sso_identities` 内のすべての行がクリアされます。ローカルアカウント自体は保持されます。
