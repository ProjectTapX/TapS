# ファイルとバックアップ

## ファイルブラウザ

インスタンス詳細ページ → **「ファイル」**タブ。

一般ユーザーは、権限のあるインスタンスのディレクトリのみ閲覧可能（`/data/inst-<short>` サブツリー。shortはインスタンスUUIDのハイフンなし先頭12桁の16進数文字）。管理者は `/data` および `/files` ツリー全体を閲覧可能です。

### 操作

| 操作 | 備考 |
|---|---|
| アップロード | チャンクプロトコル（init → chunk × N）、1 MiBごとに自動分割。各ファイルは最初にinitを実行してクォータをチェック |
| ダウンロード | ストリーミング、`Content-Disposition: attachment` を直接返却 |
| 編集 | 4 MiB以下の単一ファイルをオンライン編集可能（それより大きいファイルは読み取り専用） |
| 新規フォルダ / リネーム / コピー / 移動 / 削除 | 標準的な操作 |
| zip圧縮 | 複数の選択アイテムを1つのアーカイブにzip圧縮 |
| zip展開 | zip-slip保護を自動適用 |

### アップロードクォータ

各アップロードは**最初にinitを呼び出します**：
- Daemonが `statfs` を使用してインスタンスデータディレクトリを含むボリュームの残り容量を計算
- 申告された合計バイト数 > 残り容量の場合 → **HTTP 507 quota_exceeded**
- 成功すると `uploadId` を返却。以降のチャンクには `?uploadId=` を含める必要がある

`final=true` を送信せずにクライアントがクラッシュした場合：Daemonが1時間後に `.partial` ファイルを自動クリーンアップします。

### 既知の制限事項

- 最大単一チャンクサイズ：1 GiB（Daemon側 `MaxBytesReader`）
- 合計ファイルサイズはボリュームの残り容量に制限される
- ストリーミング圧縮/展開は非対応（圧縮時はDaemonメモリ上でzipを構築）

## バックアップ

**「バックアップ」**タブ：

| 操作 | 動作 |
|---|---|
| 作成 | インスタンス作業ディレクトリ全体をzip圧縮 → `backups/<uuid>/<timestamp>-<note>.zip` に保存 |
| 一覧 | zip名、サイズ、日時、メモを表示 |
| ダウンロード | zipをローカルにエクスポート |
| 復元 | 作業ディレクトリに展開（**同名の既存ファイルを上書き**） |
| 削除 | 単一のzipを削除 |

### バックアップ名の検証

`name` フィールドは正規表現 `^[A-Za-z0-9._-]{1,128}$` で制限されています（パストラバーサル / シェル文字を防止）。
`note` フィールドは最大512文字、改行不可。

### バックアップの保存場所

- インスタンスに**マネージドボリューム**がある場合（作成時にディスククォータを設定済み）：バックアップはそのボリューム内の `.taps-backups/` サブディレクトリに保存
- それ以外の場合はDaemonデータディレクトリ `/var/lib/taps/daemon/backups/<uuid>/` に保存

前者はボリュームクォータにカウントされます（バックアップによるディスク容量の枯渇を防止）。後者はカウントされません。

### バックアップ戦略の推奨事項

- `command` タイプのスケジュールタスクを設定 → `say` でプレイヤーに事前通知
- 時間をずらして別の `restart` タスクをcronで追加
- 重要なサーバーには、ホストレベルのスナップショット（LVM / btrfs / ZFS）を二次保護として使用 — TapSのバックアップは**アプリケーションレベル**であり、Daemonホストの障害からは保護されません
