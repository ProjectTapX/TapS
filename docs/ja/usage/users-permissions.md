# ユーザーと権限

## ロール

| Role | 説明 |
|---|---|
| `admin` | 全権限：すべてのユーザー / ノード / インスタンス / 設定にアクセス可能 |
| `user` | 一般ユーザー。デフォルトではインスタンスを参照できず、管理者による明示的な権限付与が必要 |
| `guest` | 予約済みの列挙値、**現在は無効**（ユーザーの作成・更新時は `admin` / `user` のみ受け付け、`guest` を指定すると 400 エラーが返る） |

**厳密な列挙値バリデーション**：管理者がユーザーを作成・更新する際、`role` フィールドは `admin` または `user` でなければならず、不正な値は 400 で拒否されます。

**ユーザー名/メールアドレスの大文字小文字非区別**：作成時およびログイン時に自動で `ToLower` + `TrimSpace` が適用され、SQLite では `LOWER()` ユニークインデックスを使用しています。`Admin` と `admin` は同一ユーザーとして扱われます。

## 最後の管理者の保護

- **最後の管理者を降格できない**（HTTP 400 `cannot demote the last admin`）
- **最後の管理者を削除できない**（HTTP 400 `cannot delete the last admin`）
- システムからすべての管理者が失われ、管理者ルートへアクセスできなくなることを防止します

## インスタンス権限ビット

`instance_permissions` テーブルに格納され、各行はユーザーのインスタンスに対する権限ビットマスクを記録します：

| Bit | Name | 意味 |
|---|---|---|
| 1 | `PermView` | インスタンスの参照 / モニタリング / 読み取り専用ターミナルの表示 |
| 2 | `PermControl` | 起動 / 停止 / 再起動 / 入力 + サーバーデプロイ + インスタンス設定の編集（一部フィールド） |
| 4 | `PermFiles` | ファイル管理 + バックアップ操作 + ファイルのアップロード/ダウンロード |
| 8 | `PermTerminal` | ターミナル書き込み（stdin へのコマンド送信） |
| 16 | `PermManage` | インスタンス設定の全フィールド編集（管理者相当。慎重に使用してください） |

組み合わせ可能です。例：`PermView | PermFiles = 5`、`PermAll = 31` で全権限。

**admin ロールはすべての権限チェックをバイパスします** — admin = 全権限。

## 権限付与のワークフロー

**「ユーザー管理」** → ユーザーを選択 → **「権限」** → そのユーザーの既存インスタンス権限一覧が表示されます：

1. **「権限を追加」** をクリック
2. 対象のノード + インスタンスを選択し、権限ビットにチェックを入れる
3. 保存 → 即座に反映（ミドルウェアは次のリクエスト時に DB を読み取ります）

**取り消し**：権限行の末尾にある削除ボタンをクリック → DB 行が削除 → ユーザーはそのインスタンスへのアクセスを即座に失います。

## ユーザーレベルの操作

| エンドポイント / ページ | 管理者権限が必要 | 説明 |
|---|---|---|
| ユーザー作成 | はい | 初期パスワード + ロールを設定 |
| ユーザーのパスワード変更 | はい | `tokens_invalid_before` が更新 → そのユーザーの全 JWT が即座に無効化 |
| ユーザーのロール変更 | はい | 上記と同様、既存トークンを即座に無効化 |
| ユーザー削除 | はい | そのユーザーの API キーをカスケード削除。ログイン履歴は保持。**インスタンス権限（`InstancePermission`）は現在カスケード削除されません** — 残留エントリが気になる場合は手動で取り消してください |
| ログイン履歴の表示 | はい | グローバル。IP / UA / 失敗理由を表示 |
| 監査ログの表示 | はい | グローバル。すべての POST/PUT/DELETE を記録 |
| セルフサービスパスワード変更 | ログイン済みの全ユーザー | `/api/auth/me/password` |

## セルフサービスパスワード変更

- 一般ユーザーは右上メニュー → **「パスワード変更」** からパスワードを変更できます
- `mustChangePassword=true` で初回ログインした場合、`EnforcePasswordChange` ミドルウェアによりすべての書き込みリクエストがパスワード変更ページにリダイレクトされます
- 変更後、`tokens_invalid_before` が設定 → 現在の JWT が無効化 → 再ログインが必要です

## API キーとユーザーの関係

- 各 API キーはユーザーに紐づき、**そのユーザーのロールを継承**します
- ユーザーが削除されると、そのキーはすべて**カスケード削除**されます
- **パスワード変更ではキーは自動無効化されません**（設計上の意図：パスワード変更は人間のログインを対象とし、キーは独立したライフサイクルを持つマシン認証情報です）
- キーを一括無効化するには：ユーザーが API キーページで「すべてのキーを取り消し」をクリックするか、管理者が個別に削除できます

詳細は [API キー](api-keys.md) を参照してください。

## セキュリティに関する推奨事項

- 各ユーザーに**個別のアカウント**を用意し、admin を共有しない
- admin アカウントには**長いパスワード**を使用（12 文字以上 + 複雑性）
- CI/スクリプトには **API キー**を使用し、人間のアカウントパスワードを CI 環境変数に入れない
- **ログイン履歴**を定期的に監査し、異常な IP / UA がないか確認する
- パスワードのローテーション：管理者が重要なアカウントに対して 90 日の有効期限サイクルを設定（手動リマインダー）
