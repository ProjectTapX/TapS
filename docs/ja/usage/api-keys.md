# API キー

API キーは、マシン（CI、スクリプト、外部連携）向けの長期間有効な認証情報です。人間のアカウント JWT と同じ認証ミドルウェアを共有しますが、ライフサイクルは独立しています。

## フォーマット

- 固定プレフィックス `tps_` に続く 48 文字の 16 進数（24 ランダムバイト）
- 合計長：52 文字
- 例：`tps_3fe3c349dd703a4c8b...`

## 発行

**「API キー」** ページ → **「新規キー」**：

| Field | 説明 |
|---|---|
| Name | 表示用。例：`ci-deploy` / `monitoring-prober` |
| IP Whitelist | カンマ区切りの IP または CIDR。空欄 = 任意の IP |
| Scope | カンマ区切りの権限タグ。空欄 = 全権限（ユーザーロールを継承）。下表を参照 |
| Expiration | `Never` / 30/90/365 日 / カスタム日付 |

**「OK」** をクリック → **一度だけ表示される**平文のキーが表示されます。コピーして安全に保管してください。閉じると**二度と表示されません**（DB には SHA-256 のみ保存されます）。

## スコープ（ルート権限タグ）

使用可能な値（CSV）：

| Scope | 許可される API グループ |
|---|---|
| `instance.read` | インスタンスの参照 / モニタリング / プレイヤーリスト / ノードの公開メタデータ |
| `instance.control` | 起動/停止 / 入力 / 作成 / 更新 / テンプレートデプロイ / サーバーデプロイ |
| `files` | ファイル管理 + バックアップ |
| `tasks` | スケジュールタスク |
| `admin` | ユーザー、ノード、設定、監査、API キー管理 |

空欄 = スコープ制限なし（**全権限**、ロールによる制限のみ）。

例：監視スクリプトで読み取りのみ必要 → `instance.read`、CI 自動再起動スクリプト → `instance.read,instance.control`。

## 使用方法

```bash
curl -H "Authorization: Bearer tps_3fe3c349dd703a4c..." \
     https://taps.example.com/api/instances
```

サーバー側の処理フロー：
1. `tps_` プレフィックスを検出 → API キー経路
2. SHA-256 比較 → 該当行を特定
3. IP ホワイトリストの検証（CIDR / 完全一致 IP）
4. `revoked_at` が NULL であることを検証
5. `expires_at` > 現在時刻であることを検証（NULL = 無期限）
6. リクエストのスコープ一致を検証
7. 5 回/分の失敗 → その IP を 5 分間ブロック（429 + Retry-After）

## 取り消しと削除

各キー行には 2 つのボタンがあります：

- **取り消し**（デフォルト色）：`revoked_at` を現在時刻に設定。行は監査用に保持。認証情報は即座に無効化
- **削除**（赤）：行を物理削除。復元不可

**「すべてのキーを取り消し」** ボタン：現在のユーザーの**未取り消しのすべての**キーに対して `revoked_at = now` を一括設定します。「キー漏洩の疑い」など緊急時に使用します。

## 有効期限とローテーション

- 作成時に 30 / 90 / 365 日 / カスタム日付 / 無期限を選択
- 期限切れ後、呼び出しは即座に `401 invalid api key: api key expired` を返します
- **期限切れの行は自動削除されません** — 監査用に保持。管理者/ユーザーが手動で削除してクリーンアップ可能
- CI 認証情報には 90 日サイクルを推奨。期限切れ前に新しいキーをデプロイし、古いキーは自然に期限切れにしてください

## JWT との違い

| 観点 | JWT | API キー |
|---|---|---|
| 発行 | ログイン時に署名 | ユーザーが手動で作成 |
| 取り消し | `tokens_invalid_before` の更新（全トークンを一括無効化） | キーごとの `revoked_at` フィールド（個別制御） |
| デフォルト TTL | 1 時間（設定変更可） | 無期限（作成時に任意で有効期限を設定） |
| スライディング更新 | あり（残り時間が TTL の半分未満で自動更新） | なし（固定の認証情報） |
| スコープ | なし（ユーザーロールに従う） | あり（CSV ルートグループ制限） |
| IP ホワイトリスト | なし | あり |
| 適した用途 | ブラウザでの操作 | CI / スクリプト / 監視 |

## セキュリティに関する推奨事項

- 各 CI タスクに**専用のキー**を割り当て、共有しない
- IP ホワイトリストを設定（CI ランナーの送信元 IP）
- 最小限のスコープを設定（読み取り専用の監視 → `instance.read` のみ）
- 定期的にローテーション（90 日）
- 漏洩の疑いがある場合、直ちに **「すべてのキーを取り消し」** をクリック + パスワードを変更
- キーをコードに含めたり git にコミット**しない**。GitHub Actions Secrets / GitLab CI Variables などを使用してください
